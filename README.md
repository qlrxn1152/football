# 미니프로젝트 1 

# FootMatch 1차 MVP

## 1. 1차 MVP 목표

- 회원 ( 로그인 / 회원가입 / 로그아웃 )
- 팀 ( 팀 생성 / 팀 이름변경 )
- 팀 가입 ( 특정팀에 회원이 가입요청 / 가입요청 수락 , 거절 )
- 매치 ( 특정팀이 매치를 등록 / 매치 요청을 수락 , 거절 )
- 매치 수락요청 ( 특정팀이 올라와져 있는 매치에 매치수락 요청 )
- 매치 결과에따른 팀 점수 반영

---



## 2. 왜 프로젝트를 시작 ?

프로 축구 리그를 보면서, 실제로 축구를 즐기는 사람들도 자신이 속한 팀과 함께 리그에 참여하는 듯한 경험을 느끼고 싶어할것이다라고 생각함.

또한 실제로 조기축구를 하다 보면 다음과 같은 불편함이 있습니다.

- 매치 상대 팀을 구하는 데 시간이 오래 걸림
- 상대 팀과 일정을 조율하는 과정이 번거로움
- 인원이 부족할 경우 용병을 구해야 하는 번거로움
- 경기 결과나 팀원 스탯을 수기로 관리해야 하는 불편함
- 엑셀 등으로 기록을 관리하면 직관적인 화면을 제공하기 어려움

- 회원 <-> 팀
  - ㄴ 1명의 멤버는 최대 1개의 팀에만 속할 수 있다. ( 팀에 속하지않을수도, 속한다면 1개의 팀만 )
  - ㄴ 1개의 팀에는 최소 1명의 멤버가 있어야 한다. ( 1개의 팀에는 최소 1명의 멤버, 최대 N 명의 멤버 )

---

- 멤버 <-> 팀가입신청
  - ㄴ 1명의 멤버는 최대 N 개의 팀에 가입신청을 넣을 수 있다. ( 팀이없는 멤버는, 팀에 가입신청을 안넣을수도있고 가입신청을 넣는다면 최대 N 개의 팀에 넣을수있다. )
  - ㄴ 1개의 팀 가입신청에는 무조건 1개의 회원만 있어야한다.
  - ㄴ 멤버가 가입신청을 넣게되면, status = PENDING

---

- 팀 <-> 팀가입신청
  - ㄴ 1개의 팀은 여러개의 팀가입신청을 가질 수 있다. ( 여러명의 멤버가 특정 1개의 팀에 가입신청할 수 있다. )
  - ㄴ 1개의 팀 가입신청에는 무조건 1개의 팀만 있어야한다.

---

- 팀 <-> 팀 매치
  - ㄴ 팀이 매치를 생성하게되면, 해당 팀은 자동으로 homeTeam 으로 부여된다. 또한, status = PENDING 으로 설정된다.
  - ㄴ 1개의 팀은 여러개의 팀 매치를 가질 수 있다.
  - ㄴ 1개의 팀 매치는 1개의 홈팀 / 1개의 어웨이팀을 가질 수 있다.

---


- 팀매치 <-> 팀 매치 요청
  - ㄴ 특정팀이 매치를 생성하고 또 다른팀이 매치요청을 보낸경우, match_request_status = PENDING, 매치를 생성한 팀이 아닌 다른팀은 자동으로 awayTeam 으로 부여된다.
  - ㄴ 1개의 팀 매치는, 여러개의 팀 매치 요청을 받을 수 있고, 매치 요청이 없을 수 있다. ( 최대 N 개, 최소 0개 )
  - ㄴ 1개의 팀 매치 요청은, 1개의 홈팀 / 1개의 어웨이팀을 가질 수 있다.

---

- 팀매치 <-> 팀 매치 결과
  - ㄴ 1개의 팀 매치는 최대 1개의 팀 매치 결과를 받을 수 있다. ( 즉, 매치결과가 아직 없을수도있고, 입력해도 1개의 매치에는 1개의 결과만 존재한다. )
  - ㄴ 1개의 팀 매치 결과는 무조건 1개의 팀 매치를 가져야한다.

## 4.핵심 기능

### 회원

- 회원가입
- 로그인
- 로그아웃

### 팀

- 팀 생성
- 팀 이름 변경
- 팀 목록 조회
- 팀 상세 조회

### 팀 가입 신청

- 회원은 특정 팀에 가입 신청을 보낼 수 있다.
- 팀장은 가입 신청을 수락할 수 있다.
- 팀장은 가입 신청을 거절할 수 있다.

### 매치

- 팀장은 자신의 팀으로 매치를 등록할 수 있다.
- 다른 팀의 팀장은 등록된 매치에 매치 요청을 보낼 수 있다.
- 매치를 등록한 팀의 팀장은 들어온 매치 요청을 수락할 수 있다.
- 매치를 등록한 팀의 팀장은 들어온 매치 요청을 거절할 수 있다.

### 경기 결과

- 매치가 성사되면 경기 결과를 입력할 수 있다.
- 경기 결과에 따라 팀 점수가 반영된다.
- 승리 팀은 점수가 증가한다.
- 패배 팀은 점수가 감소한다.
- 무승부인 경우 양 팀의 점수가 반영된다.

## 5. 핵심 도메인

FootMatch의 핵심 도메인은 다음과 같습니다.

- Member
- Team
- TeamJoinRequest
- TeamMatch
- TeamMatchRequest
- TeamMatchResult

각 도메인은 다음 흐름을 중심으로 연결됩니다.

```text
회원가입 / 로그인
→ 팀 생성 또는 팀 가입 신청
→ 팀장이 가입 신청 수락
→ 팀장이 매치 등록
→ 다른 팀장이 매치 요청
→ 매치 등록 팀장이 요청 수락
→ 경기 결과 입력
→ 팀 점수 반영
```

## 6. 현재까지의 핵심 정책 ( 검증 )

## 팀 정책
- 회원은 팀을 생성할 수 있다.
- 팀을 생성한 회원은 해당 팀의 팀장이 된다.
- 이미 팀에 소속된 회원은 새로운 팀을 생성할 수 없다.
- 이미 팀에 소속된 회원은 다른 팀에 가입 신청을 할 수 없다.
- 팀 이름은 중복될 수 없다.

## 팀 가입 정책
- 팀에 가입하지 않은 회원만 가입 신청을 보낼 수 있다.
  - ㄴ 가입신청을 보내면, TeamJoinRequest.status = PENDING
- 팀장은 가입 신청을 수락하거나 거절할 수 있다.
  - ㄴ 수락시에는, TeamJoinRequest.status = ACCEPTED
  - ㄴ 거절시에는, TeamJoinRequest.status = REJECETD
- 가입 신청이 수락되면 회원은 해당 팀의 팀원이 된다.


## 매치 정책
- 팀장만 매치를 등록할 수 있다.
  - ㄴ 매치 등록하면, TeamMatchStatus.status = PENDING
- 팀장만 등록된 매치에 매치 요청을 보낼 수 있다.
- 자신의 팀이 등록한 매치에는 요청을 보낼 수 없다.
- 이미 요청한 매치에는 중복 요청을 보낼 수 없다.
- 매치를 등록한 팀의 팀장만 매치 요청을 수락하거나 거절할 수 있다.
  - ㄴ 수락시에는, TeamMatchStatus.status = ACCEPTED -> AwayTeam 설정
  - ㄴ 거절시에는, TeamMatchStatus.status = REJECETD

## 경기 결과 정책
- 성사된 매치에 대해서만 경기 결과를 입력할 수 있다.
- 경기 결과 입력 후 매치 상태는 완료 상태로 변경된다.
  - ㄴ TeamMatchStatus.status = COMPLETED
- 경기 결과에 따라 팀 점수가 변경된다.

## 해당 프로젝트 용어사전

- 점수 : `rating`
- PK : `테이블명_id`
- 검증 메서드 이름 : `validateForXxx` ( Xxx : 무엇을 검증하는지 )
  - ㄴ 예시 : 팀 리더인지 체크 => `validateForTeamLeader`
- 홈팀 아이디 : `home_team_id`
- 어웨이팀 아이디 : `away_team_id`
- 상태 : `테이블명_status`
  - ㄴ 예시 : 매치 상태 => `team_match_status`
  - ㄴ 예시 : 가입신청 상태 => `team_request_join_status`
  - ㄴ 예시 : 매치 요청 상태 => `team_match_request_status`
